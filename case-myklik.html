Overview

End-to-end redesign of a low-code / no-code platform that allows users to build their own systems based on databases — with the goal of making both the application-building process and database management as intuitive as possible for non-technical teams.

01 — Challenge

A system no one
could use alone

The existing platform had significant usability problems. The interface lacked clear instructions, making it inaccessible to anyone without prior training — and even the project owner struggled to complete certain tasks without external support.

My first step was to review the current solution and map the key pain points in the database creation process. A usability test with a first-time user quickly confirmed what the heuristic review suggested: the entire workflow was unclear, from onboarding to publishing a first application.

Three main flows structured the redesign: system access and onboarding, database creation and management, and the Studio — where the application interface is actually built. Each flow required a different level of complexity, but all shared the same fundamental problem: the product spoke in technical language to non-technical people.

Key Finding — Usability Test

"She needs training to understand each button and what it does — the platform feels easy to navigate, but impossible to learn alone."

01.1 — Heuristic Review

Where the
interface failed

A usability test was conducted with a company employee who had never used the system before. Her session was recorded and the transcript analysed against Nielsen's 10 heuristics. The findings confirmed critical violations across 5 heuristics — all pointing to the same root cause: the interface assumed knowledge the user didn't have.

Critical
H1 — Visibility of System Status
The user had no indication of where she was in the flow or what step came next. She repeatedly asked "how do I proceed to the next step?" — the system offered no progress cues or confirmation of completed actions.
Critical
H6 — Recognition Rather Than Recall
Every button required prior knowledge to interpret. The user needed training before she could understand what each control did — nothing was self-evident. Icons lacked labels, and actions were not described in context.
Major
H2 — Match Between System and Real World
The interface used technical terminology unfamiliar to non-developer users. The user struggled to understand the purpose of screens and fields because the language did not match how she thought about her own work.
Major
H3 — User Control and Freedom
The user did not know how to close open windows and expected them to dismiss when clicking elsewhere — a standard interaction pattern that was not supported. Dead ends were frequent, with no clear escape route.
Moderate
H4 — Consistency and Standards
Interactions varied across sections without clear reason. Some panels closed on outside-click, others did not. The same action had different affordances in different contexts, breaking the user's mental model of how the system behaved.
Moderate
H10 — Help and Documentation
No inline guidance, tooltips, or contextual help existed anywhere in the platform. The user's only recourse when stuck was to ask someone — making independent use of the system effectively impossible for new users.

02 — Process

From confusion
to clarity

Discovery
Heuristic Review + Interview
Reviewed the existing interface and conducted an in-depth interview with a company employee who had never used the system before — identifying which interactions were intuitive and which caused friction.
Define
3 Core Flows
Structured the product around three main journeys: system access and onboarding, database creation and management, and the Studio — where the app interface is assembled. Each required a tailored approach.
Benchmarks
Market & Pattern Analysis
Studied how comparable platforms — CRM tools, spreadsheet software, app builders — structured their interfaces. The goal was to identify familiar interaction patterns that could reduce the learning curve without oversimplifying.
Design
High-Fidelity Screens
Translated the research into detailed high-fidelity designs — applying real UI elements, color, and typography. Covered profile settings, subscription management, app customization, and the workflow automation editor.
03 — Solution

Design
principles

The final phase focused on the Studio — a builder environment where users assemble the application interface by adding inputs, buttons, forms, and other UI elements. The most complex challenge was designing the workflow automation editor: a system where users configure trigger-based actions without writing code.

For example, if a user fills in a field and leaves it blank, the system should automatically create an account and send a password. To make this approachable, I drew on conditional logic patterns already familiar from other contexts — mapping them to visual, language-based rules using if/else statements rather than code syntax.

Principle 01
Familiar patterns first
The interface was modelled after dashboards users already know — CRM tools, spreadsheet software. Familiarity reduces anxiety. When in doubt, design for recognition, not learning.
Principle 02
Structure before visual
Core flows and information architecture were defined before any high-fidelity work began. Usability consistency with the product's principles always came before aesthetics.
Principle 03
Logic without code
Workflow automation was designed around conditional rules expressed in plain language — if/else statements anyone can read — making complex behaviour accessible to non-technical users.

04 — Results

The
outcome

Despite the constraints — the client chose not to include real data in the onboarding and opted against an interactive prototype — the high-fidelity screens delivered a solid foundation for implementation.

By aligning the interface with familiar patterns and industry standards, the redesign made the system more approachable to users with varying levels of technical expertise. The project was successfully approved, and its implementation provided a more intuitive and structured experience for building applications.

Takeaway

"Designing for non-technical users isn't about removing power — it's about removing the fear of using it."
